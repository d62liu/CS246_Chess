\section{Overview}
% ─────────────────────────────────────────────

This document describes the design and implementation plan for a command-line chess program supporting human and computer players, text and graphical display, and a setup mode for custom board configurations.

The architecture follows a layered design with clear separation of concerns:

\begin{itemize}
    \item \textbf{Controller layer} (\code{ChessProgram}): Reads and dispatches commands from standard input. Handles the \code{game}, \code{move}, \code{resign}, \code{undo}, and \code{setup} commands.
    \item \textbf{Game logic layer} (\code{Game}): Manages game state including turn tracking, scoring, and end-condition detection (checkmate, stalemate, resignation).
    \item \textbf{Board layer} (\code{Board}): Maintains the 8$\times$8 grid of pieces, validates and executes moves, and implements all chess rules including castling, en passant, and pawn promotion. Inherits from \code{Subject} to notify display observers after each move.
    \item \textbf{Piece layer} (\code{ChessPiece} and its six subclasses): Each piece knows its candidate moves. The \code{Board} filters these into legal moves by checking paths, boundaries, friendly-fire, and king safety.
    \item \textbf{Strategy layer} (\code{Strategy} and four concrete levels): Encapsulates computer AI decision-making, from random moves to minimax search.
    \item \textbf{Display layer} (\code{TextDisplay}, \code{GraphicDisplay}): Observers attached to the board that re-render after every state change.
\end{itemize}

All layers are compiled as C++ modules (using \code{export module}), which enforces interface boundaries and reduces compilation coupling compared to traditional header includes. 
\subsection{Changes from Due Date 1}

The original UML design included \code{Player}, \code{HumanPlayer}, and \code{ComputerPlayer} class hierarchies, as well as \code{PieceFactory}, \code{StrategyFactory}, and \code{PlayerFactory} classes. During implementation, I found these abstractions to be unnecessary:

\begin{itemize}
    \item \textbf{Player hierarchy removed.} Human moves are simply read from \code{stdin} by \code{ChessProgram::handleMove()}, and computer moves are obtained by calling \code{Strategy::pickMove()}. A \code{PlayerType} enum suffices to distinguish the two cases, avoiding an entire class hierarchy that would have added complexity without meaningful polymorphic benefit.
    \item \textbf{Factory classes replaced by factory functions.} The free functions \code{makePiece()} and \code{makeStrategy()} serve the same purpose as the originally-planned factory classes, but with less boilerplate. Since each factory is a single function with a \code{switch} statement, wrapping them in classes would not have improved cohesion or testability.
    \item \textbf{Observer interface simplified.} The original design had \code{Observer::notify(Position, Piece*)}, passing the changed square explicitly. In practice, the displays simply re-render the entire board (which is fast for an 8$\times$8 grid), so a parameterless \code{update()} method proved cleaner.
    \item \textbf{Memory management upgraded to STL smart pointers.} The original design used raw owning pointers for the board grid and move history. The final implementation uses \code{std::shared\_ptr<ChessPiece>} throughout, with no explicit memory-management statements anywhere in the source. This is described in detail in the Extra Credit section.
\end{itemize}

\section{Design}
% ─────────────────────────────────────────────

\subsection{Design Patterns}

\subsubsection{Observer Pattern (Board $\to$ Displays)}

\code{Board} inherits from \code{Subject}, which maintains a list of \code{Observer*} pointers. After every call to \code{movePiece()}, \code{undoMove()}, \code{placePiece()}, or \code{removePiece()}, the board calls \code{notifyObservers()}, which invokes \code{update()} on each attached observer. Both \code{TextDisplay} and \code{GraphicDisplay} implement \code{Observer::update()} by reading the board state and rendering it.

This decouples the board from any knowledge of how it is displayed. Adding a new display (e.g., a networked display or a move-logging observer) requires only writing a new \code{Observer} subclass and attaching it. No changes to \code{Board}, \code{Game}, or \code{ChessProgram} are needed.

The \code{setNotify(bool)} method on \code{Board} allows temporarily disabling notifications, which is used during computer move evaluation (when the strategy performs speculative \code{movePiece}/\code{undoMove} calls that should not trigger display updates).

\textbf{Code location:} \code{src/game/display/observer/observer.h} (interface), \code{src/game/board/board.cc} (notification calls).

\subsubsection{Strategy Pattern (Computer AI)}

The abstract class \code{Strategy} declares a single pure virtual method \code{pickMove(Board\&, Color)}. Four concrete strategies implement this:

\begin{itemize}
    \item \textbf{Level1Strategy:} Selects a uniformly random legal move.
    \item \textbf{Level2Strategy:} Scores each move by material captured (+10 per piece value) and whether it gives check (+5), then picks the highest-scoring move with random tiebreaking.
    \item \textbf{Level3Strategy:} Extends Level 2 by penalizing moves where the moved piece lands on a square attacked by the opponent ($-8$ per piece value at risk).
    \item \textbf{Level4Strategy:} Performs a 2-ply minimax search. For each candidate move, it simulates the opponent's best response and evaluates the resulting material balance. Checkmate is scored at $+100{,}000$; stalemate at $0$.
\end{itemize}

The factory function \code{makeStrategy(PlayerType)} returns a \code{std::unique\_ptr<Strategy>} for the requested level. \code{ChessProgram::handleMove()} creates a strategy on each computer turn and calls \code{pickMove()}.

\textbf{Code location:} \code{src/chess\_strategy/strategy.h} (interface), \code{src/chess\_strategy/strategy.cc} (all four levels).

\subsubsection{Factory Functions (Piece and Strategy Creation)}

\code{makePiece(PieceType, Color, Position)} returns a \code{std::unique\_ptr<ChessPiece>} for the requested piece type. This centralizes piece construction and is used by \code{Board::initBoard()}, \code{Board::placePiece()}, and pawn promotion logic.

\code{makeStrategy(PlayerType)} returns a \code{std::unique\_ptr<Strategy>} for the requested difficulty level. This is used by \code{ChessProgram::handleMove()} when it is a computer player's turn.

Both functions follow the Factory Method pattern in function form --- the client code depends only on the abstract base class, not on concrete subclass constructors.

\textbf{Code location:} \code{src/game/pieces/pieces.h} (\code{makePiece}), \code{src/chess\_strategy/strategy.h} (\code{makeStrategy}).



\subsection{Memory Ownership Model}

All heap-allocated chess pieces are owned through \code{std::shared\_ptr<ChessPiece>}. The board grid is declared as \code{std::shared\_ptr<ChessPiece> grid[8][8]}. The \code{Move} struct's \code{captured} and \code{promoted\_from} fields are also \code{std::shared\_ptr<ChessPiece>}.

This design exploits the dual role of \code{Move}:

\begin{itemize}
    \item In the \textbf{legal moves vector} (before a move is executed), \code{Move::captured} holds a shared reference to the piece still in the grid. The grid retains co-ownership; reference count is 2.
    \item In the \textbf{history vector} (after a move is executed), the grid slot for the captured piece is set to \code{nullptr}. The \code{shared\_ptr} in \code{Move::captured} becomes the sole owner; reference count drops to 1.
\end{itemize}

When \code{undoMove()} restores a captured piece to the grid, it assigns the \code{shared\_ptr} back into the appropriate grid slot. When the \code{Move} is subsequently popped from history and destroyed, the reference count drops to 1 (owned only by the grid once more).

Non-owning pointers are kept as raw \code{ChessPiece*} where appropriate: \code{Move::piece\_moved} and \code{Move::rook\_moved} always point to pieces whose lifetime is managed by the grid and express no ownership. \code{Subject::observers} stores non-owning \code{Observer*} pointers --- each observer's lifetime is managed externally (e.g., \code{ChessProgram::run()} holds the \code{GraphicDisplay} in a \code{std::unique\_ptr<GraphicDisplay>}).

\textbf{Code location:} \code{src/game/board/board.h} (struct and class definitions), \code{src/game/board/board.cc} (all operations).

\subsection{Cohesion and Coupling}

Each module has a single, well-defined responsibility:

\begin{itemize}
    \item \code{types} --- value types and enums only; no logic.
    \item \code{pieces} --- piece movement rules; no board or game knowledge.
    \item \code{board} --- grid state, move validation, and rule enforcement; no UI or command parsing.
    \item \code{game} --- turn management, scoring, and end-condition detection; delegates move execution to \code{Board}.
    \item \code{chess\_program} --- command parsing and dispatch; delegates all game logic to \code{Game}.
    \item \code{strategy} --- AI decision-making; reads board state but never modifies it permanently (uses \code{movePiece}/\code{undoMove} pairs for speculative evaluation).
    \item \code{observer}, \code{text\_display}, \code{graphic\_display} --- display rendering; depend on \code{Board}'s read-only interface.
\end{itemize}

Coupling is minimized by the module system: each module imports only what it needs. The dependency graph is acyclic: \code{types} $\leftarrow$ \code{pieces} $\leftarrow$ \code{board} $\leftarrow$ \code{game} $\leftarrow$ \code{chess\_program}, with \code{observer} and display modules branching off from \code{board}. \code{strategy} depends on \code{board} and \code{types} but not on \code{game} or \code{chess\_program}.

\section{Resilience to Change}
% ─────────────────────────────────────────────

\subsection{Adding a New Piece Type}

A new piece type (e.g., for a chess variant) requires:
\begin{enumerate}
    \item Adding a value to the \code{PieceType} enum.
    \item Creating a new subclass of \code{ChessPiece} with \code{get\_valid\_moves()} and \code{symbol()}.
    \item Adding a case to \code{makePiece()}.
\end{enumerate}

No changes are needed to \code{Board}, \code{Game}, \code{ChessProgram}, or the display classes, because they operate on the \code{ChessPiece} abstract interface.

\subsection{Adding a New AI Difficulty Level}

\begin{enumerate}
    \item Create a new \code{Strategy} subclass (e.g., \code{Level5Strategy}).
    \item Add a \code{Computer5} value to \code{PlayerType}.
    \item Add a case to \code{makeStrategy()} and \code{parsePlayerType()}.
\end{enumerate}

The \code{Game} and \code{Board} classes are unaffected.

\subsection{Adding a New Display Type}

\begin{enumerate}
    \item Create a new \code{Observer} subclass (e.g., \code{NetworkDisplay}).
    \item Attach it to the board in \code{ChessProgram::run()}.
\end{enumerate}

No changes to \code{Board}, \code{Game}, or existing displays are required.

\subsection{Changing the Board Representation}

The \code{Board} class encapsulates its internal representation (currently \code{std::shared\_ptr<ChessPiece> grid[8][8]}). If we wanted to switch to a \code{std::unordered\_map<Position, std::shared\_ptr<ChessPiece>>} or a flat \code{std::array}, only \code{Board}'s private implementation would change. All external code accesses pieces through \code{getPiece(Position)}, which returns a non-owning \code{ChessPiece*}.

\subsection{Supporting a Different Game (e.g., Four-Handed Chess)}

The \code{Board} class would need to be extended to support a larger grid (e.g., a cross-shaped 14$\times$14 board used in four-player chess). The \code{Color} enum would gain two more values. \code{Game} would track four players and cycle turns among them. The \code{Strategy} classes would need updated evaluation functions. However, the Observer pattern, the Strategy pattern, the piece hierarchy, and the module structure would all remain intact --- the architectural patterns are game-agnostic.

\section{Answers to Questions}
% ─────────────────────────────────────────────

\subsection{Opening Book}

A book of standard openings could be implemented as a trie (prefix tree) keyed by move sequences. Each node stores the set of ``known good'' responses with associated weights (e.g., from historical win/draw/loss statistics). At the start of a game, the computer player would consult the trie: if the current move history matches a prefix in the book, the computer selects from the book's suggested responses (weighted randomly or by win rate). Once the game leaves the book (no matching prefix), the computer falls back to its normal strategy.

Implementation-wise, this would be a new class \code{OpeningBook} that wraps a \code{std::map<std::vector<Move>, std::vector<WeightedMove>>}. The \code{Strategy} subclasses would check the opening book first before performing their normal evaluation. This requires no changes to \code{Board} or \code{Game} --- only \code{Strategy} would be extended, consistent with the open/closed principle.

This answer is unchanged from my Due Date 1 response.

\subsection{Undo}

Undo is fully implemented. The \code{Board} class maintains a \code{std::vector<Move> history} where each \code{Move} records:

\begin{itemize}
    \item The piece moved and its previous \code{has\_moved} state.
    \item Any captured piece (held as a \code{std::shared\_ptr<ChessPiece>}) and its position (which differs from the destination for en passant).
    \item Castling metadata: the rook's from/to positions and its previous movement state, with the rook identified by a non-owning \code{ChessPiece*}.
    \item Promotion metadata: \code{promoted\_from} holds the original pawn as a \code{std::shared\_ptr<ChessPiece>}, removed from the grid and replaced by the promoted piece.
\end{itemize}

Calling \code{Board::undoMove()} pops the last entry from the history and reverses all state changes. \code{Game::undoMove()} additionally switches the turn back. Unlimited undos are supported. The history is cleared when a new game starts.

On Due Date 1, I proposed storing a full board snapshot per move. The final implementation stores only the delta (the \code{Move} struct), which is more memory-efficient and avoids the complexity of deep-copying the entire board state. This differs from my Due Date 1 answer, which described a snapshot-based approach.

\subsection{Four-Handed Chess}

To support four-handed chess:

\begin{enumerate}
    \item \textbf{Board geometry.} Four-player chess uses a cross-shaped board (typically 14$\times$14 with corners removed). The \code{Board} class would use a \code{std::array<std::array<std::shared\_ptr<ChessPiece>, 14>, 14>} with a static validity mask indicating which cells are in bounds.
    \item \textbf{More colors.} The \code{Color} enum would expand to four values (e.g., White, Black, Red, Blue).
    \item \textbf{Turn order.} \code{Game} would cycle through four players; the current \code{turn} field would rotate through all four colors.
    \item \textbf{Win conditions.} Under free-for-all rules, a player whose king is captured is eliminated and their pieces removed. The last survivor wins. Under team rules (White+Red vs.\ Black+Blue), the game ends when both kings of one team are out.
    \item \textbf{Strategy updates.} AI evaluation would generalize to max-n minimax or paranoid search.
    \item \textbf{Display.} Both text and graphical displays would render a larger, non-rectangular board and distinguish four piece colors.
\end{enumerate}

The Observer pattern, Strategy pattern, and piece hierarchy carry over without structural changes. This answer is unchanged from my Due Date 1 response.

\section{Extra Credit Features}
% ─────────────────────────────────────────────

\subsection{Memory Management via STL Containers and Smart Pointers}

The program contains no explicit memory-management statements anywhere in the source. All heap-allocated chess pieces are managed exclusively through \code{std::shared\_ptr<ChessPiece>} and \code{std::unique\_ptr<ChessPiece>}. 

\textbf{What changed and why \code{shared\_ptr} was chosen:}

The \code{Board} grid was converted from a raw pointer array (with manual cleanup in the destructor and several helper methods) to \code{std::shared\_ptr<ChessPiece> grid[8][8]}. The \code{Move} struct's \code{captured} and \code{promoted\_from} fields became \code{std::shared\_ptr<ChessPiece>}.


\textbf{What remained as raw pointers:}

Non-owning raw \code{ChessPiece*} pointers are retained where the spec explicitly allows them: \code{Move::piece\_moved} and \code{Move::rook\_moved} always point to pieces whose lifetime is managed by the grid. \code{Subject::observers} stores non-owning \code{Observer*} pointers whose lifetime is managed by the caller.

\textbf{Temporary grid rearrangement:}

\code{wouldLeaveInCheck()} and \code{givesCheck()} temporarily rearrange pieces in the grid to simulate a move, check king safety, then restore the original state. With raw pointers this was a simple reassignment; with \code{std::shared\_ptr} it requires \code{std::move} to shift ownership between grid slots without inadvertently destroying pieces. The solution saves temporarily-displaced pieces into local \code{shared\_ptr} variables (\code{savedTo}, \code{savedEP}), performs the simulation, then moves them back:

\begin{lstlisting}
auto savedTo = std::move(grid[to.getX()][to.getY()]);
// ... simulate move ...
grid[to.getX()][to.getY()] = std::move(savedTo); // restore
\end{lstlisting}

The \code{Board} destructor was removed entirely as the compiler-generated destructor correctly destroys the \code{shared\_ptr} array and \code{vector<Move>} members, which in turn release all piece memory automatically.

\textbf{Code location:} \code{src/game/board/board.h}, \code{src/game/board/board.cc}.

\section{Final Questions}
% ─────────────────────────────────────────────

\subsection{Lessons Learned About Writing Large Programs}

This project taught me the value of establishing clear interfaces early. By defining module boundaries and abstract base classes (\code{ChessPiece}, \code{Strategy}, \code{Observer}) upfront, different components could be developed independently.

I also learned that the initial design rarely survives contact with implementation unchanged. The Player class hierarchy from our Due Date 1 design proved to be over-engineered; simplifying it to an enum led to cleaner code. Being willing to deviate from the original UML when a simpler solution presented itself was an important lesson in pragmatic design. Working alone meant I had to manage scope carefully and prioritize a working core over ambitious features.

Finally, incremental testing proved essential. Building the command interpreter and text display early allowed us to test each piece type and board rule as it was implemented, catching bugs before they could compound. This was especially important for edge cases like en passant interacting with the undo mechanism.

\subsection{What We Would Do Differently}

If starting over, I would:

\begin{itemize}
    \item \textbf{Write more tests from the beginning.} A more comprehensive suite of unit tests for \code{Board::movePiece()} and the strategy classes would have saved significant debugging time in finding and resolving edge cases.
    \item \textbf{Implement the graphical display earlier.} I saved it for last, which meant debugging X11 integration under time pressure. Building it alongside the text display would have distributed that effort more evenly.
    \item \textbf{Design the \code{Move} struct with smart pointers from the start.} Converting from raw owning pointers to \code{std::shared\_ptr} later in development required careful reasoning about every place where pieces are temporarily removed from the grid. Designing with smart pointers from the beginning would have made ownership invariants clearer throughout development and avoided the need for a late-stage refactor.
\end{itemize}

\section*{Conclusion}
% ─────────────────────────────────────────────

My chess program implements all required features: full chess rules (castling, en passant, pawn promotion), check/checkmate/stalemate detection, human and computer players at four difficulty levels, text and graphical displays, setup mode with validation, unlimited undo, and score tracking. The design makes effective use of the Observer, Strategy, and Factory patterns, and the C++23 module system enforces clean separation between components. All memory is managed through STL smart pointers with no explicit memory-management statements in the source. The architecture is resilient to change: new pieces, strategies, and displays can be added with minimal modification to existing code.
